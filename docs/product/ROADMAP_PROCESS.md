# Roadmap Process

Forge roadmap work is organized through milestones, epics, features, and releases.

## Planning Units

| Unit | Meaning |
| --- | --- |
| Milestone | A coherent product goal or platform phase. |
| Epic | A major capability inside a milestone. |
| Feature | A user-visible or platform capability with acceptance criteria. |
| Ticket | A specific implementation or documentation task. |
| Release | A tagged delivery package with notes and validation. |

## Roadmap Lifecycle

1. Identify operational problem.
2. Draft feature specification.
3. Validate against product principles.
4. Check Project Sentinel applicability.
5. Decide whether an RFC is required.
6. Decide whether an ADR is required.
7. Plan implementation, testing, documentation, and release notes.
8. Deliver through pull request and Definition of Done.
9. Update roadmap and release history.

## Prioritization Criteria

Prioritize work that:

- Reduces administrative burden.
- Improves human review and auditability.
- Makes exercise context clearer.
- Improves operational asset relationships.
- Supports Project Sentinel validation.
- Strengthens Forge Academy and documentation.
- Preserves modular architecture.

## Roadmap Review

Roadmap changes should consider:

- User role impact.
- Operational value.
- Architecture impact.
- Security impact.
- Documentation impact.
- Training impact.
- Regression and reference exercise impact.
