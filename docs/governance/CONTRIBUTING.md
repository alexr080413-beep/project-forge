# Contributing To Forge

Forge is developed as a disciplined open-source platform for Exercise Control workflows.

## Development Workflow

1. Read relevant repository documentation.
2. Open or reference an issue.
3. Create a focused branch.
4. Make the smallest clear change.
5. Add or update tests.
6. Update documentation.
7. Run relevant checks.
8. Open a pull request using the template.

## Branch Naming

Use:

```text
codex/<milestone>-<ticket>-<short-topic>
```

Examples:

```text
codex/governance-001-project-foundation
codex/atlas-003-knowledge-graph
```

## Commit Style

Use clear, imperative commits:

```text
docs: add governance foundation
feat: add exercise data engine
fix: correct review queue status
test: cover knowledge graph payload
```

## Issue Process

- Use the bug report template for defects.
- Use the feature request template for new capabilities.
- Use the documentation template for docs gaps.
- Include documentation impact in every issue that affects workflows, setup, models, APIs, plugins, or architecture.

## Pull Request Process

Pull requests should include:

- Summary.
- Tests run.
- Documentation updates.
- Human review and security considerations.
- Definition of Done validation.

## Review Expectations

Reviewers should check:

- Correctness.
- Test coverage.
- Documentation coverage.
- Forge Academy impact.
- Human review principle.
- Security posture.
- UI consistency.
- Architecture impact.
- Release note impact.
